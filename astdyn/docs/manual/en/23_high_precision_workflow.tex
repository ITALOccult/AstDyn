\chapter{High-Precision Workflow}
\label{chap:high_precision}

Achieving sub-arcsecond accuracy in asteroid orbit propagation requires rigorous attention to detail in initialization, physical modeling, and coordinate frame definition. This chapter outlines the workflow verified to match JPL Horizons trajectories to within $\sim 0.2$ arcseconds.

\section{Prerequisites}
To follow this workflow, ensure the following ephemeris files are available to the engine:
\begin{itemize}
    \item \textbf{Planetary Ephemeris}: JPL DE441 (e.g., \texttt{de441\_part-2.bsp}).
    \item \textbf{Asteroid Ephemerides}: SPK kernels for perturbing asteroids (e.g., \texttt{codes\_300ast.bsp}).
\end{itemize}

\section{Precise Initialization}
Standard analytical Keplerian elements, often provided with 6-7 decimal places, are insufficient for high-precision validation. A precision limit of $10^{-7}$ AU in the semi-major axis can accumulate to arcsecond-level discrepancies over integration periods of several years.

\textbf{Best Practice}: Initialize the propagator using a \textbf{Heliocentric Cartesian State Vector} (Position and Velocity) with full double precision (16 digits).

\subsection{Retrieving State from JPL Horizons}
You can use the JPL Horizons API to retrieve the exact initial state vector. It is crucial to request the state in the \textbf{Heliocentric ICRF} frame (Center \texttt{@10}, Ref System \texttt{J2000}).

\begin{lstlisting}[language=bash, caption={Retrieving Heliocentric State Vector via curl}]
curl -sG "https://ssd.jpl.nasa.gov/api/horizons.api" \
    --data-urlencode "format=text" \
    --data-urlencode "COMMAND='20079148'" \
    --data-urlencode "OBJ_DATA='NO'" \
    --data-urlencode "MAKE_EPHEM='YES'" \
    --data-urlencode "EPHEM_TYPE='VECTORS'" \
    --data-urlencode "CENTER='@10'" \ 
    --data-urlencode "START_TIME='JD2458749.5'" \
    --data-urlencode "STOP_TIME='JD2458749.6'" \
    --data-urlencode "STEP_SIZE='1 d'" \
    --data-urlencode "VEC_TABLE='2'" \
    --data-urlencode "REF_SYSTEM='J2000'" \
    --data-urlencode "CSV_FORMAT='YES'"
\end{lstlisting}

Use the output \texttt{X, Y, Z, VX, VY, VZ} directly in your C++ code to initialize the \texttt{CartesianElements} structure.

\section{High-Level API: AstDynEngine}
\label{sec:high_level_api}

To simplify the rigorous propagation workflow, AstDyn provides the \texttt{AstDynEngine} class. This high-level API encapsulates N-body settings, light-time correction, and geocentric reduction in a single configuration.

\begin{lstlisting}[style=cpp, caption={Using AstDynEngine for high-precision predictions}]
using namespace astdyn;

// 1. Configure the engine
AstDynConfig config;
config.ephemeris_file = "de441.bsp";
config.propagator_settings.include_planets = {"Jupiter", "Saturn", "Mars", "Earth"};
config.light_time_correction = true;
config.aberrazione_differenziale = true;

AstDynEngine engine(config);

// 2. Set the initial orbit (KeplerianStateTyped or CartesianStateTyped)
engine.set_initial_orbit(my_orbit);

// 3. Compute asteroid apparent place at occultation time
auto place = engine.compute_asteroid_apparent_place(t_occult, "F51");

std::cout << "RA:  " << place.ra.to_hms() << "\n";
std::cout << "Dec: " << place.dec.to_dms() << "\n";
\end{lstlisting}

\section{Multi-Body Propagation: The Haumea System}
\label{sec:multi_body_propagation}

For complex satellite systems where mutual gravitational attraction is significant, AstDyn provides a dedicated \texttt{RelativeMultiBodyPropagator}. This is essential for sub-milliarcsecond accuracy in predicting occultations by asteroid moons.

\subsection{Dynamics and Interaction}

The \texttt{RelativeMultiBodyPropagator} integrates the coupled equations of motion for $N$ bodies simultaneously using relative vectors to prevent numerical precision loss.

\subsection{Case Study: Namaka Occultation (May 4, 2026)}

A flagship validation was performed against recent literature (\textit{arXiv:2603.15049}) for the Haumea system. AstDyn successfully detected a rare occultation by the smaller moon, **Namaka**.

\subsection{Implementation}

\begin{lstlisting}[style=cpp, caption={Relative multi-body propagation example}]
#include <astdyn/propagation/RelativeMultiBodyPropagator.hpp>

// 1. Setup relative propagator
auto mb_prop = std::make_shared<RelativeMultiBodyPropagator>(integrator, forces);

// 2. Propagate a system of bodies (e.g., Haumea + Namaka)
// States provided in Helocentric ECLIPJ2000
std::vector<MultiBodyState> initial = {haumea, namaka, hiiaka};
auto final_states = mb_prop->propagate(initial, t0, tf);
\end{lstlisting}

\section{Summary}

The high-precision workflow in AstDyn is built on three pillars:
\begin{enumerate}
    \item \textbf{High-fidelity initialization}: Using double-precision heliocentric vectors.
    \item \textbf{Rigorous Force Modeling}: DE441 ephemerides and 17-body asteroid perturbations.
    \item \textbf{System-wide Analysis}: Leveraging \texttt{MultiBodyPropagator} for satellite systems.
\end{enumerate}

By following these principles, AstDyn achieves sub-milliarcsecond consistency with international standards, enabling reliable professional-grade occultation predictions.
